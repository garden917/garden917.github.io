Summary

규제·보안·현장 운영·파트너 구조처럼 제약이 큰 도메인에서, 현장의 복잡한 사정을 업무 흐름과 데이터 기준으로 풀어온 TPM/PM입니다. 보험 플랫폼에서는 상담신청→설계사 배정→계약→정산 흐름을, 제조업에서는 주문→배정→방문→설치/AS→재고/수불 흐름을, 인테리어 플랫폼에서는 리드 품질·파트너 운영·민원 데이터를 요금제와 운영 판단 기준으로 연결했습니다.

제 강점은 제품·플랫폼 구축·전환·고도화 과제에서 기능 범위와 실행 기준을 실제 사용자가 더 효율적으로 움직일 수 있는 흐름으로 연결하는 데 있습니다. 누가 어떤 데이터를 입력·확인하고 그 데이터가 어디에 남아 다음 업무로 이어지는지, 어떤 권한·보안·운영 기준이 필요한지를 함께 정리해 전체 제품·시스템 흐름이 자연스럽게 연결되게 만듭니다.

핵심 역량

1. 제품·플랫폼 과제를 사용 흐름으로 구체화합니다

토스인슈어런스 CRM 사용률 하락은 고객관리 화면 문제가 아니라, 고객정보가 CRM 밖으로 빠져나가고 상담·계약 퍼널 데이터가 비는 문제로 봤습니다. 세인홈시스에서는 고객관리 화면 수준이 아니라 주문·재고·상담·설치/AS가 한 흐름으로 이어지는 통합 운영관리 시스템으로 범위를 넓혀 정리했습니다. AI UW는 보험사별 약관·인수기준표를 어떤 데이터 구조로 읽고 어떤 질문까지 답할 수 있는지 검증하는 PoC로 잡았습니다.

2. 흩어진 업무를 끝까지 이어지는 흐름으로 만듭니다

토스인슈어런스에서는 토스앱 상담신청→CRM 고객정보 수신→설계사 배정/상담→ERP 계약 저장→CMS 수수료 정산으로 이어지는 흐름을 기준으로 CRM과 ERP/CMS를 봤습니다. ERP/CMS에서는 인사·조직·계약·상품·수수료·리스크가 정산으로 모이는 구조를 시스템 로직과 데이터 싱크 기준으로 연결했습니다.

3. 백오피스와 현장 앱의 역할을 나눕니다

세인홈시스에서는 본사가 주문·재고·상담·배정을 관리하는 통합 운영관리 시스템과, 기사가 방문·설치 완료·사진·서명·QR/GPS 기록을 남기는 현장 서비스 앱을 나눠 설계했습니다. 현장에서 남긴 데이터가 다시 본사 AS, 재고, 민원 처리에 쓰이게 한 경험은 이후 설계사앱/SFA에서 외부 활동 중 고객 확인·서류 발송·전자서명·상담 입력을 돕는 모바일 운영 인프라로 이어졌습니다.

4. 보안·데이터 조건을 쉬운 판단 기준으로 풀어냅니다

AI UW PoC에서는 정형 상품 데이터는 RDB·Text-to-SQL로, 약관·인수기준 같은 문서는 Vector DB·RAG로 다루는 기준을 만들었습니다. NPS CRM 연동에서는 고객 식별값을 쓸 수 있는지, 법인 간 데이터 제공 근거와 동의/처리방침·Kafka 연동 흐름이 맞는지를 나눠 정리했습니다. CMS 내재화에서는 외주 클라우드와 ASP 종료 리스크를 내부 IDC, 소스코드 매입, 장애 대응 책임 문제로 풀었습니다.

5. 지표를 실제 액션으로 연결합니다

전환율을 하나의 숫자로 보지 않고 상담신청→배정→컨택→계약 중 어디서 빠지는지 봅니다. 집닥에서는 CAC·LTV·ROI를 요금제와 프로모션 운영 방침으로 연결했고, 토스인슈어런스에서는 리드타임·유효 퍼널·골든타임을 컨택 타이밍, DB 배분, 후속 분석 과제로 연결했습니다.
