프로필

ERP, CMS, CRM, FSM, AI 업무 시스템처럼 여러 조직과 시스템이 얽힌 환경에서 목적, 범위, 의존성, 리스크, 출시 후 운영 기준을 설계해 온 9년차 Technical Program Manager입니다. 단순 일정 관리보다, 사업 요구와 사용자 행동을 제품 관점에서 해석하고 개발팀·외주사·현업·보안·운영 조직이 같은 기준으로 실행하도록 만드는 데 강점이 있습니다.

제 강점은 무엇을 만들지만 정하는 것이 아니라, 왜 지금 이 문제가 중요한지, 어떤 기술·운영 제약이 있는지, 어떤 범위부터 잘라야 실제로 출시되고 운영되는지 판단하는 일입니다. 플랫폼을 내부 도구가 아니라 사업·운영·데이터 흐름을 바꾸는 제품 시스템으로 다룹니다.

Core Strengths

1. 복잡한 기술 프로그램의 목적·범위·의존성을 정리합니다

토스인슈어런스 ERP 도입에서는 입사 직후 흩어진 의사결정 히스토리를 역추적해 도입 목적, 1차 오픈 범위, TF 구성, 보안·운영 우선순위를 정리했습니다. 이후 RFP, SRS, 데이터 마이그레이션 기준, 주간보고, 장애 대응 체계를 통해 60명 규모 TF가 같은 기준으로 움직이도록 만들었습니다.

2. 기술 리스크를 사업·운영 리스크로 번역합니다

CRM 사용률 95%→64.5% 하락을 단순 사용성 문제가 아니라 고객정보 유출, 상담 퍼널 분석 공백, DB 활용 로드맵 붕괴로 이어지는 영업 데이터 인프라 리스크로 정의했습니다. ERP-CRM 강결합, 보안 인증, 마스킹, 앱 정책 대응, 외주 ASP 종료처럼 기술 이슈가 사업 운영에 미치는 영향을 기준으로 우선순위를 잡았습니다.

3. 외주사와 내부 조직이 같은 실행 기준으로 움직이게 합니다

ERP, CMS, 설계사 앱, 세인홈시스 제조 ERP/CRM/FSM처럼 외주사와 내부 조직이 함께 움직이는 프로젝트에서 요구사항, 계약, 산출물, 테스트, 유지보수 기준을 직접 정리했습니다. 세인홈시스에서는 OMS/WMS/TMS 어드민과 FSM 앱 범위를 SRS, 메뉴/데이터셋, 마이그레이션 양식, 테스트 시나리오로 구체화했습니다.

4. 플랫폼을 제품처럼 판단합니다

내부 플랫폼을 단순 운영 도구로 보지 않고, 사용자 행동과 데이터 흐름을 바꾸는 제품으로 봅니다. CRM v2에서는 고객 리스트, 고객 상세, 보장분석, 네비게이터, 엑셀, 어드민을 영업 퍼널 기준으로 정렬했고, 상담신청→버퍼계정→설계사배정→고객컨택→ERP계약의 5단계 퍼널을 정의했습니다.

5. 출시 이후 운영까지 프로그램 범위로 봅니다

ERP 오픈 이후 4년+ 운영 안정화, 월 100억+ 수수료 정산 CMS 자동화, 지넥슨 ASP 종료 이후 내부 개발 채널 기반 이슈 접수, 긴급 대응, 현업 공지 체계 전환까지 관리했습니다. 설계사 앱은 출시 이후 Android 14/15, Google Play 삭제 위기, 정책·기관 API 대응까지 운영 범위에 포함했습니다.
