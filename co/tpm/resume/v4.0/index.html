요약

규제·보안·현장 운영·파트너 구조처럼 제약이 큰 도메인에서, 여러 조직과 시스템이 얽힌 제품·플랫폼 구축·전환·고도화 과제를 맡아 온 9년차 TPM/PM입니다. 보험 플랫폼의 ERP/CMS/CRM/SFA와 AI UW PoC, 제조업 통합 운영관리 시스템처럼 레거시 전환, 정산 자동화, 현장 운영, 외주·보안 제약이 함께 맞물린 프로젝트를 주로 다뤘습니다.

업무 프로세스 매핑을 통해 기능 목록보다 실제 처리 순서와 부서 간 인계 지점을 먼저 정리합니다. 보험 플랫폼에서는 상담신청→설계사 배정→계약→정산 흐름을, 제조업에서는 주문→배정→방문→설치·AS→재고·수불 흐름을, 인테리어 플랫폼에서는 리드 품질·파트너 운영·민원 데이터를 요금제와 운영 판단 기준으로 연결했습니다.

전사 시스템 담당자로 30~60명 규모 TF에서 현업·외주·보안·Core 조직과 함께 일했습니다. 범위 관리, 요구사항 정의, 이해관계자 정렬, 데이터 이관, RFP·SRS·WBS, 테스트·UAT·오픈 기준, 외주 관리, 출시 후 안정화까지 전 과정을 조율했습니다. 요구가 불명확할수록 목적·범위·위험·성공 기준을 먼저 정리하고, 출시 이후에도 장애, 권한, 정책 변경, 데이터 품질을 운영 범위로 관리합니다.

제 강점은 복잡한 현업 요구를 실행 가능한 제품 기준으로 바꾸는 데 있습니다. 누가 어떤 데이터를 입력·확인하고, 그 데이터가 어디에 남아 다음 업무로 이어지는지 정리한 뒤 권한·보안·품질·운영 기준을 붙여 제품과 플랫폼 전체가 자연스럽게 이어지게 만듭니다.

핵심 역량

1. 업무 흐름을 매핑하고 문제를 다시 정의합니다

토스인슈어런스 CRM 사용률 하락을 화면 문제가 아니라 고객정보가 외부 도구로 빠져나가고 상담·계약 데이터가 비는 문제로 재정의했습니다. 세인홈시스에서는 고객관리 요청을 주문·재고·상담·설치·AS가 이어지는 통합 운영관리 범위로 확장했습니다. AI UW는 약관·인수기준표를 어떤 데이터 구조로 읽고 어떤 질문까지 답할 수 있는지 검증하는 PoC로 정의했습니다.

2. 범위와 요구사항을 여러 조직의 실행 기준으로 바꿉니다

전사 시스템 담당자로 30~60명 규모 TF에서 현업·외주·보안·Core 조직과 함께 일했습니다. 사업 목적과 제약을 1차 오픈 범위, 구현 우선순위, 데이터 이관, RFP·SRS·WBS, 테스트·UAT·오픈 기준으로 구체화해 여러 조직이 같은 기준으로 움직이게 했습니다.

3. 데이터 정합성과 시스템 의존성을 함께 관리합니다

토스앱 상담신청→CRM 고객정보 수신→설계사 배정·상담→ERP 계약 저장→CMS 수수료 정산 흐름을 기준으로 시스템 간 데이터와 의존성을 정리했습니다. ERP의 계약·상품·조직·인사 월 마감 데이터를 CMS 계산 기준으로 연결하고, 원수사 대사부터 SAP·회계 수기 반영까지 금액 차이를 역추적할 수 있는 검수 기준을 정리했습니다. 세인홈시스에서는 본사 백오피스와 현장 앱의 역할을 나눠 설치·증적·재고 데이터가 AS와 민원 처리로 돌아오게 만들었습니다.

4. 외주·보안·권한 제약을 비교 가능한 판단 기준으로 바꿉니다

AI UW PoC에서는 RDB·Text-to-SQL과 Vector DB·RAG를 나누고 비개인정보 PoC 범위와 본계약 전환 시 리전·KMS·권한 조건을 분리했습니다. CMS 내재화는 외부 ASP 유지, 소스코드 매입, 내부 재구축을 비용·보안·운영권·장애 책임 기준으로 비교했습니다. NPS CRM 연동은 고객 식별값, 법인 간 데이터 제공 근거, PDP 승인, Kafka 수신 흐름을 실제 구현 전 조건으로 정리했습니다.

5. 흐름별 지표를 우선순위와 다음 행동에 연결합니다

전환율을 하나의 숫자로 보지 않고 상담신청→배정→컨택→계약 중 어느 단계에서 이탈하는지 분석합니다. 집닥에서는 CAC·LTV·ROI를 요금제와 프로모션 정책으로 연결했고, 토스인슈어런스에서는 리드타임·유효 퍼널·골든타임을 컨택 시점, DB 배분, 후속 분석 과제로 연결했습니다.
