Program 01

보험 ERP·수수료 정산·자동차보험 비교견적 시스템 일괄 구축 총괄: 레거시 전환과 정산 자동화

전사 시스템 일괄 구축 총괄 · ERP·정산·비교견적 · 60명 TF

기존 수수료 솔루션은 신규 데이터를 저녁 8시 이후에만 등록할 수 있고 토스인슈어런스가 필요한 계산을 지원하지 않아, 여러 팀이 Excel로 계산한 결과를 저장하는 수준이었습니다. ERP·CMS·자동차보험 비교견적 시스템을 한 일정으로 구축하면서 ERP 월 마감 데이터를 CMS 계산 기준으로 연결하고, 여러 팀의 정산·시상·환수 규칙과 검증 기준을 시스템화했습니다.

약 5개월·20주ERP 구축 기간과 주간보고
60명 TF현업·외주·Core·보안
3,500명·60만 건사용자·계약 데이터
10일→3일월 100억+ CMS 정산

판단과 실행

  • 범위 관리: 보안 취약 레거시 교체와 기존 운영 유지를 최우선으로 두고, 1차 오픈을 MYGA 대체와 기존 CRM 유지로 제한했습니다. 수수료·인사·계약·리스크·보안·인프라는 별도 실행 축으로 나눴습니다.
  • 요구사항 정의: 수수료팀의 기본 정산·법인시상·본사 시책과 리스크·컴플라이언스팀의 퇴직·해약 환수 요구를 SRS와 화면정의서에 연결하고, 조직 7단계·직급 8단계가 수수료 로직과 함께 동작하도록 구체화했습니다.
  • 이해관계자 정렬: 내부 30명, 외주 10명, Core·보안 20명이 같은 목적과 오픈 기준으로 움직이도록 RFP·SRS·WBS·테스트·마이그레이션 기준을 관리했습니다.
  • 예외 처리 기준: 기본 산식 밖의 수금이관·월중 조직개편·복수 계약 해지 및 퇴직자 환수·신규 상품 첫 정산을 귀속·배분·공제 순서·검수 주체로 나눠 시스템 로직과 승인 절차로 정의했습니다.
  • 데이터 정합성: 3개년 데이터를 정제·이관하고 ERP의 계약·상품·조직·인사 월 마감 데이터를 CMS 계산 기준으로 연결했습니다. 원수사 대사부터 SAP·회계 수기 반영까지 금액 차이를 역추적하는 순서도 정리했습니다.
  • 검증 기준: 순번 1번 모집자 고정·모집자 수수료율이 수금자보다 낮은 입력 차단·배분 합계 100% 미달 차단, 외주 개발 결과와 회계·수수료팀 실제 데이터 검수, 35개 원수사 정산액과 내부 계산액 대사를 세 단계로 분리했습니다. 대사 대상 금액이 100% 일치해야 월 마감을 승인하도록 했습니다.
  • 프로세스 자동화: 3명이 엑셀로 처리하던 월 정산을 CMS로 전환해 인력 변동 없이 처리기간을 10일에서 3일로 줄였습니다.
  • 트레이드오프 분석: 외부 ASP 유지, 소스코드 매입, 내부 재구축을 비용·보안·지식재산권·운영권·장애 책임 기준으로 비교해 단계적 내재화 방향을 정했습니다.
  • 출시 후 안정화: 초기 월 80건 이상의 변경 요청과 약 10건의 장애를 분류하고 외주 정례 운영과 장애 대응 기준을 세워 월 50건 이하·장애 5건 이하 수준으로 낮췄습니다.

결과

  • ERP·CMS·자동차보험 비교견적 3개 시스템을 공통 일정 안에서 구축했고, 2022년 12월 오픈한 ERP는 2026년 기준 사용자 3,500명·계약 데이터 60만 건 규모의 운영 기반으로 확장됐습니다.
  • CMS 월 정산 사이클은 10일에서 3일로 단축했고, ERP 정산 배치는 후속 안정화를 거쳐 현재 4시간 이내 처리 가능한 수준이 됐습니다.
  • 휴먼에러·재정산·재마감은 월 10건 이상에서 2건 내외로 줄었습니다. 35개 원수사 대사 대상 금액의 100% 일치를 월 마감 승인 기준으로 적용했습니다.

Program 02

보험 CRM v2 재구축: 고객 데이터 수신·적재 개선과 상담 UI 재설계

CRM v2 재구축 · 고객 데이터·상담 UI · 30명+ TF

CRM 사용률 하락을 고객정보 유출, 상담 과정 데이터 공백, 회사 DB 자산 손실 위험으로 정의하고 CRM v2 재구축을 추진했습니다. 프론트 재구축과 고객정보 수신 정확도 개선, 영업 상태 관리, 시스템 의존성 분리를 함께 진행한 프로젝트입니다.

64.5%→90%CRM 활성 사용률
30명+·3개월TF와 초기 릴리즈
약 1,000명산정 당시 전체 설계사
180만/17만19개월 상담·계약 분석

판단과 실행

  • 문제 정의: 설계사가 Trello·Notion·Excel 같은 외부 도구로 이탈하는 현상을 고객정보 보호와 회사 데이터 보존, 상담 퍼널 분석이 동시에 무너지는 문제로 다시 정의했습니다.
  • 데이터 요구사항: CRM을 잘 쓰는 설계사 인터뷰, 실제 컬럼, 데이터 1만 건을 함께 받아 사용 맥락과 필요한 필드를 대조했습니다.
  • 데이터 정합성: 토스코어 마이데이터에서 받는 가입 보험 정보의 정확도와 수신 속도를 개선하고 상담에 필요한 고객 상세·보장분석 정보를 다시 구성했습니다.
  • 의존성 관리: ERP 장애가 CRM으로 이어지는 강결합을 낮추고 SSO 통합과 VPN→2FA 전환으로 외부 활동 중에도 사용할 수 있는 접속 구조를 만들었습니다.
  • 퍼널 분석: 토스앱 상담신청→고객정보 수신→설계사 배정·상담→ERP 계약 저장→CMS 정산의 5단계를 정의하고 CRM·ERP 데이터를 연결해 상담량과 계약의 관계를 분석했습니다.
  • 지표 정의: 기능 사용량만 보지 않고 상담 상태 변화, 고객 여정의 연결성, 사용자 유형별 행동을 후속 판단 지표로 정리했습니다.
  • 단계적 전환: S3 파일 암호화는 일부 서버에서 먼저 검증한 뒤 전체로 확대해 데이터 전환 위험을 낮췄습니다.

결과

  • 2023년 약 1,000명 설계사 기준 CRM 활성 사용률을 64.5%에서 90%로 회복해 고객 데이터가 회사 시스템 안에 남는 비중을 높였습니다. 이 변화는 계약 전환 상승의 인과 효과와 구분됩니다.
  • 19개월 상담 180만 건·계약 17만 건 분석에서 상담량만 늘린다고 계약이 비례해 증가하지 않음을 확인하고, 리드 품질과 배정 기준을 함께 판단할 근거를 만들었습니다.
  • 규제 변화로 수요가 급증한 구간에는 타깃 리스트와 상담 스크립트를 2일 만에 배포했지만, 해당 월 증가분을 기능 단독 성과로 귀속하지 않았습니다.

Program 03

보험 AI 언더라이팅 서비스 도입 PoC: 보험 지식 검색 구조와 도입 조건 검증

AI 언더라이팅 도입 PoC · 검색·벤더·보안 조건 검증

세일즈 서포트팀이 반복 처리하던 보험 가입·보장 문의를 AI로 옮길 수 있는지 검증하는 PoC를 기획·설계했습니다. 모델 기능을 먼저 고르지 않고 질문 유형, 원천 데이터, 정답 판단자, 검색 경로, 보안 조건, PoC 범위와 확장 조건을 먼저 정리했습니다.

5개월요구사항 정의~계약 준비
일 40건·30분기존 UW 문의 업무량
50→30→10문항검증 질문 정제
35개 보험사인수기준표 대응 범위

판단과 실행

  • 요구사항 정의: 문의를 가입 가능 여부, 보장 가능 여부, 신규 상품 문의로 나누고, 답변에 인수기준표·약관의 근거 페이지와 줄을 제시하도록 검증 조건을 정했습니다.
  • 외주 관리: 국내 AI 솔루션 6개사를 검토하고 3개사를 상세 비교한 뒤, 비정형 원천 처리 정확도와 운영 사용성을 기준으로 후보를 좁혔습니다.
  • 데이터 요구사항: 비교·집계가 필요한 정형 상품 데이터는 RDB, 약관·인수기준 같은 서술형 문서는 메타데이터를 붙인 Vector DB로 나눴습니다.
  • 범위 관리: 질문 의도에 따라 정형 질의는 Text-to-SQL, 서술형 질의는 RAG로 보내는 하이브리드 검색 구조를 PoC 범위로 정의했습니다.
