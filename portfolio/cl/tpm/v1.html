01

멀티 팀 프로그램 구조화

복잡한 0→1 프로젝트를 입사 첫 주에 구조화합니다. 보험 ERP 도입 시 60명 규모 TF(개발·네트워크·DBA·DevOps·보안·QA·운영)를 직접 조직했고, 6단계 로드맵을 설계해 1차 오픈 범위를 "MYGA 대체 + 기존 CRM 유지"로 잘라냈습니다. "보안 이슈 해결(1순위) > 기존 운영 유지(1순위) > 사용성 개선(2순위)"를 입사 3일 차에 명문화하고, RFP·SRS·WBS·주간보고 가이드까지 첫 두 달 안에 일괄 세팅했습니다.

02

시스템 간 의존성 관리

ERP·CMS·CRM·FSM·SAP가 동시에 움직이는 환경에서 시스템 간 인터페이스와 데이터 정합성을 설계합니다. 보험 ERP는 인사·조직·계약·상품·수수료·통계가 분리·수기로 운영되던 상태를 통합 ERP로 묶었고, ERP-CMS-SAP 데이터 싱크 구조로 월 100억 원 이상 수수료 정산을 자동화했습니다. 3개년 데이터 마이그레이션은 항목/입력/기간 × 조직정보·인사정보·계약·사용인코드·상품목록·조정률 매트릭스로 기준을 표준화했습니다.

03

외주·계약 프로그램 설계

외주·내부 R&R을 운영 가능한 체계로 만듭니다. 지넥슨 ASP 도입 직후 협업 프로토콜 5개(태스크 전달·메일 vs 슬랙·상주·장애 대응·월말 보고)를 설계했고, 유지보수 계약을 0.5M/월 풀에서 0.25M/월 + 별도 프로젝트 구조로 재설계했습니다. 노임단가 협상은 4단계 구조로 근거를 확보했습니다. 이 프로토콜은 이후 SAP(BSG)·영선아이티 등 후속 외주 운영 패턴으로 그대로 재사용됐습니다.

04

기술 리스크·장애 통제

인프라·보안·장애 의사결정을 직접 합니다. ERP는 k8s 기반 컨테이너, Redis 캐시, 로그 분리·이중화 인프라에 AD·마스킹·VPN 보안을 내재화했습니다. 보안 코드리뷰의 Path Traversal HIGH를 우선순위로 분류하고, GC Storm 장애는 캐싱·CDN·부하 테스트 3단계로 재발 방지를 설계했습니다. 4년+ 운영에서 장애 재발률 0%를 유지했습니다.

05

사용자·운영 데이터 기반 판단

기술 의사결정의 근거로 사용자 흐름과 운영 데이터를 직접 봅니다. CRM 사용률 95%→64.5% 하락은 단순 UX 문제가 아니라 고객정보 유출·상담 퍼널 분석 공백·DB 활용 로드맵 붕괴로 이어지는 영업 데이터 인프라 리스크로 재정의했습니다. 백로그를 정렬하기 전에 상담 DB 187만 건, 계약 DB 4.2만 건을 직접 분석해 조직별 계약율과 리드타임 가설을 재검증했고, 그 결과로 30명+ TF 3개월 릴리즈의 우선순위를 잡았습니다.

프로그램 구조화 의존성 관리 외주·계약 리스크·장애
상황

보안팀이 "수정·미티게이션 보완 불가" 판정한 MYGA 시스템(PHP 5.2 + 그누보드 2016)이 핵심 보험 운영을 담당하고 있었고, 인사·조직·계약·상품·수수료·통계가 별도 시스템 + 엑셀로 분리·수기 관리되는 상태였습니다. 외주(지넥슨) 도입 결정만 존재하고 Why/Who/What/How는 어디에도 정리되어 있지 않았습니다.

제품 판단

1차 오픈 범위를 "MYGA 대체 + 기존 CRM 유지"로 잘라냈습니다. 수수료·위해촉·인사·계약·리스크·보안·인프라는 별도 트랙으로 분해했고, 보안 이슈 4개(자동차 비교견적·디아모 KMS·스마트확인서플러스·모바일 ERP)를 "지금/나중/협의"로 다시 분해해 1차 범위에 들어갈 항목만 골랐습니다.

실행

60명 TF 직접 조직, 6단계 로드맵 설계, RFP·SRS·WBS·주간보고 가이드 일괄 세팅. ASP vs SI를 굿케이스/배드케이스 시나리오로 판단해 ASP 채택. 3개년 데이터 마이그레이션 기준 매트릭스 표준화. k8s + Redis + 로그 이중화 + AD/마스킹/VPN 인프라·보안 내재화. ASP 협업 프로토콜 5개 설계 + 유지보수 계약 0.5M→0.25M+별도 프로젝트 구조로 재설계.

근거
  • 사용자 3,500명·계약 60만 건 규모 운영 무중단 신규 오픈 (2022.12 1차 오픈)
  • 4년+ 안정 운영, 장애 재발률 0%

Case 4 · 토스인슈어런스 · 2022 – 현재

정산 CMS 자동화 + ASP 종료 로드맵

의존성 관리 외주·계약 리스크·장애
상황

월 100억 원 이상 수수료 정산이 엑셀 수작업 기반이었고, ERP-CMS-SAP 간 데이터 싱크가 비표준이었습니다. 5년간 누적된 외주 구조적 문제가 ASP 의존을 만성화시키고 있었습니다.

제품 판단

정산 자동화와 외주 의존 종료를 별도 트랙이 아니라 같은 로드맵으로 봤습니다. 명세서 CRM 내재화 → CMS 수수료 기능 → ERP 완전 내재화 단계로 분해했습니다.

실행

ERP-CMS-SAP 데이터 싱크 구조 설계. 수수료율 검증 로직 3개(순번 고정 / 비교 / 합산 100%)로 휴먼에러를 시스템 차단. 5년간의 외주 구조적 문제를 증적 기반으로 정리해 ASP 종료와 소스코드 매입 협상 근거로 활용. 지넥슨 ASP 종료 후 내부 개발 채널 기반 이슈 접수·긴급 대응·현업 공지 체계로 운영 전환.

근거
  • 월 100억 원 이상 수수료 정산 자동화
  • 엑셀 수작업 제거, ERP-CMS-SAP 데이터 싱크 자동화
  • ASP 종료 + 소스코드 매입 협상 진행
Case 5 · 토스인슈어런스 · 인켑 + AI UW 센터 · 2024 – 현재

AI 도입 분기 운영

프로그램 구조화 리스크·장애 데이터 기반 판단
상황

AI 활용 요구가 여러 부서에서 동시에 발생하고 있었고, 사내 IT 문의 자동화와 외부 설계사용 판단 지원 등 성격이 다른 요구가 한 트랙으로 묶이려는 상황이었습니다.

제품 판단

하나의 AI 트랙으로 묶지 않고, 인켑은 IT Manager 전산 문의 자동화 + 운영 품질 관점으로, AI UW 센터는 외부 설계사용 판단 지원 PoC + 솔루션 비딩·보안·견적·데이터 구조 관점으로 분리해서 관리했습니다.

실행
  • 인켑: Slack 연동, 데이터셋 오류 수정, 웹 확장 검토, 관리자 권한 정리. 수수료·계약팀 응대 확장 검토.
  • AI UW 센터: 국내 6개 AI 솔루션 컨택, 3개사 체크, 로민 등 후보 검토 후 올거나이즈 선정. 약관·인수기준표·상품 FAQ를 RDB + 벡터DB 기반 하이브리드 Agent로 구성. 가입 가능 여부 판단 정확도 80% 이상 기준, 보안 검수, PoC 진행 조건 정리.
근거
  • 인켑 도입 후 IT Manager 반복 문의 자동화
  • AI UW 센터 PoC 진행, 보안·견적·데이터 구조 검수 통과
  • AI 도입을 도구 단위가 아니라 업무 흐름·데이터 구조·권한·품질·검증 루프 안에 넣는 패턴 정착

Working Style

판단을 빠르게 내리되, 그 판단이 운영 가능한 형태로 남도록.

  • —요청을 바로 개발 항목으로 바꾸지 않고 문제의 사업적 의미부터 정의합니다.
  • —비용·리스크·사용자 가치·기술 제약을 함께 놓고 우선순위를 정합니다.
  • —데이터가 기존 가설과 다르면 가설을 바꾸고, 이미 정해진 해법도 재검증합니다.
  • —개발자처럼 말하려 하기보다, 개발자가 판단하기 쉬운 선택지와 우선순위를 정리합니다.
  • —외주와 내부 조직의 책임 경계를 문서화하고, 회의체·보고·장애 대응·유지보수 기준을 운영 체계로 만듭니다.
  • —AI는 도구 도입보다 목적별 업무 흐름·데이터 구조·권한·품질·검증 루프 안에 넣는 것이 중요하다고 봅니다.
