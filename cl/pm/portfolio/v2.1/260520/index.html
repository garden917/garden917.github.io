§ 01다섯 가지 패턴 The Five Patterns

슬랙 업무 이력 184건 · 회사 4곳을 정독해 추출한 의사결정 휴리스틱.

P1

비용 대비 가치부터 본다

기능 요청을 받으면 가장 먼저 "이 기능이 정말 필요한가, 누가 얼마나 자주 쓸 것인가"를 묻습니다. 솔루션 도입이 default가 아닙니다.

"막상 구현했는데 안쓰는 기능이면 비용써서 개발할 필요가 없을 것 같아서 좀 더 체크하도록 할게요!" — 케이스 D 원문

P2

개발 안 하고 풀 수 있으면 안 한다

문제를 받으면 먼저 "시스템 수정 없이 운영 방식 조정으로 풀 수 있는가"를 검토. 단기 임팩트보다 기술부채를 우선합니다.

"상담내용만 넣고 상태값은 대기로 해서 설계사에게 전달하면 신규 DB로 찍혀서 나가겠네용??" — CRM 신규 DB 표기, 개발 공수 0 해결

외주는 계약서·견적서·증적으로

"비용 오른 근거 무엇인지? 수렴할 수 없다. 프로젝트 시작 당시 인건비 기준으로 계산되어야 한다." — 지넥슨 노임단가 협상

보안·운영·비용 트레이드오프가 얽힌 의사결정은 항상 3개 이상의 옵션을 표로 정리. "조용히 하나 고르지 마"라는 본인 룰의 행동 패턴.

"Option 1: 망연계 / Option 2: RPA 내재화 / Option 3: 에이전시프로 유지" — CAR 원수사 데이터 수신, 3옵션 비교

영역 경계 넘어 끼어든다

§ 03케이스 — 패턴의 증거 Cases as Evidence

서술이 아니라 패턴의 증거로 압축했습니다. 각 케이스 우측 상단의 색 점은 어떤 패턴이 작동했는지 표시합니다.

C01 · 집닥 · 2019–2020

만성 적자 플랫폼 BM을 unit economics로 재정의 + 3년치 민원 분류 체계 재설계

기능 개선이 아니라, 어디에 손대야 사업 구조가 바뀌는지를 판단했습니다. 동시에 운영 데이터로 분류 체계 자체를 재정의했습니다.
BM 문제를 CAC·LTV·ROI·채널별 성과·파트너별 전환율로 나눠 봤고, DB 등급제·파트너 등급제로 좋은 리드가 좋은 파트너에 배정되는 구조를 설계. 단기 매출 이벤트가 아니라 BEP 운영 체계로 정렬. 같은 시기에 3년치 계약 5,388건·민원 1,010건·세분화 1,877건을 직접 분석해 민원 분류 체계(유형 6·강도 3·공종 17·귀책 2)를 재설계하고 CMS 반영을 제안. 11번가·코리아빌드 2019·카이로스 등 외부 제휴와 부분공사 신사업 견적계산기까지 같은 호흡으로 운영. 이 unit economics·민원 분석·외부 제휴 감각은 후일 CRM 퍼널 분석·계약 분류 체계의 base가 됨.

C02 · 세인홈시스 · 2021–2022

제조 현장 ERP/CRM/FSM을 하나의 운영 시스템으로 + 발주자 측 외주 거버넌스

대리점·기사·본사·CS가 같은 데이터·프로세스로 움직이게 만드는 것이 운영 제품 PM의 일. 외주 산출물에 끌려가지 않고 발주자 측에서 SRS·계약·유지보수를 직접 챙긴 패턴은 토스 ASP 종료의 base가 됨.
ERP/CRM/FSM을 각각 기능이 아니라 고객 접수→기사 배정→방문→설치/수리→완료→재고 수불의 하나의 운영 퍼널로 정의. QR/GPS/알림톡/모바일 서명으로 현장 데이터가 자연스럽게 쌓이는 장치 설계. 외주 선정에서 평가항목 17개(전문분야·개발수준·기업 위치·견적금액·포트폴리오·납기·규모·특장점·단점·커뮤니케이션 등)를 직접 정해 25개 업체 → 10개 미팅 → 5개 평가 → 1개 선정(에이앤드, 6,900만 원, 2021.03~06) 진행. SRS·메뉴/데이터셋·마이그레이션 수식 레이아웃·테스트 시나리오를 직접 작성. 1년 유지보수 계약(연 850만 원)을 직접 협상·체결 (계약서 담당자 "민정원 프로" 명시, 2021.11). LG헬로비전 그린싱크 B2B AS 운영 데이터까지 직접 관리. 토스에서 설계사 앱·ERP·CRM 설계 + ASP 종료 협상 base가 됨.

C03 · 토스인슈어런스 · 2023–2025

CRM v2 + 리드타임 분석 환경 셋업과 분석 로드맵

요청받은 화면을 고치는 게 아니라, 그 뒤에 있는 사업 리스크를 발견하고 분석할 수 있는 환경까지 PM이 직접 설계.
CRM v2 — 사용률 하락을 영업 운영 인프라로 (P2)
사용률 95%→64.5% 하락을 단순 UX 문제가 아니라 고객 정보 유출·상담 퍼널 분석 마비·DB 활용 로드맵 붕괴로 정의. 30+ TF와 3개월 안에 CRM v2 릴리즈. 일부 운영 이슈는 시스템 수정 없이 운영 방식 조정만으로 해결(P2 — 신규 DB 표기 누락은 CRM 로직 분석 후 운영팀 업무 방식 변경 제안으로 개발 공수 0 해결). 180만 상담·17만 계약·19개월 데이터로 "상담 물량 = 계약 성장" 전제 검증.
리드타임 분석 환경 셋업 — 분석 가능 조건을 PM이 직접 열음 (P5)
분석 요청을 받은 뒤 쿼리 결과만 기다리지 않고, 분석이 가능해지는 전제(권한·계정·DB 접근)를 PM이 직접 셋업. #core-토인슈-data 채널을 열며 코어 DE Head에게 AD/시스템 계정·ERP/CRM DB 접근 권한 요청, CMS DB 접근은 별도 스레드로 분리해 지넥슨 측 팔로업, DA의 Sqoop·Jenkins 권한 막힘은 IT 매니저에게 직접 요청해 새벽 5시 배치 시간까지 확정.
매칭 경로·정의 직접 설계 (P4)
D'amo 암호화 우회: ERP scp.DEC_B64('GNX256', perno)를 하둡에서 쓸 수 없어 서버 개발자에게 복호화 배치 테이블 dw.ntis_contract_matching 신설 요청, 이중 매칭 기준 설계(이름+전화 / 이름+생년). 사업단 정보 우회 조인: com_user_info.counselor_group이 오염 → ERP tbin_empmst → tbin_scd 2단계 우회 경로 발굴. 컨택일 정의: process_status_cd='00034' 초안 → 8/27 배포 로직 변경 + 9/3 재배정 시 컨택일 null 초기화 운영 이슈를 직접 정리·공유 → 최종 ub_customer_counsel_info 조인 + MIN(reg_dt)로 기준 변경. 취소/삭제 분류, V3 이후로 분석 범위 결정(2020년 이상 데이터 발견 + CPO 의견 반영).

C04 · 토스인슈어런스 · 2022–

ERP/CMS — 운영 시스템 구축 + 수수료 환수 도메인 모델 미스매치

불명확한 대형 프로젝트의 목적·범위·외주 협업·운영 기준을 먼저 만들고, 매월 깨지는 이유를 외주 솔루션 표준 모델과 운영 도메인 모델의 구조적 충돌로 정의.
1차 범위와 실행 인프라 (P2)
1차 범위를 MYGA 대체 + 기존 운영 유지 + 보안 리스크 해소로 자름. 60명 TF가 같은 기준으로 움직이도록 RFP·SRS·주간보고·외주 협업 프로토콜·장애 대응 체계 설계.
회계 무결성 시스템 (P1)
CMS는 정산을 회계 무결성·내부 통제 문제로 보고 ERP-CMS-SAP 데이터 싱크·IFRS 자동 산출·2022년 기말잔액 마이그레이션 검증 기준까지 연결. 월 100억+ 수수료 정산을 시스템으로 흡수.
수수료 환수 도메인 모델 미스매치 — 근본 문제 3가지 직접 정의 (P3 + P4)
매월 수수료 검증에서 4일씩 소요되는 패턴이 누적되는 동안, #product-erp-cms-car-인슈 채널에서 수수료팀·리스크팀·수수료 관리자·지넥슨 4개 그룹과 12회 리플 스레드 직접 주도(2023.10.26~11.07)로 원인 분해.
구조적 충돌: 지넥슨 표준(기지급비례 — 이미 지급한 수수료 × 환수율)은 단순하고 예측 가능. 토스인슈(수입비례 — 본인수입금액 / 총수입금액 점유율)는 다층 수수료 관계 반영 가능. 새 보험 상품·수금이관·OR(오버라이드)·복수 계약 해지 때마다 분모·귀속 기준이 흔들려 매월 수동 개입과 수정 개발 발생.

§ 05잘 맞는 역할 Best Fit

위 5개 패턴이 동시에 작동해야 하는 환경에서 가장 효과적입니다.

①

복잡한 B2B/B2B2C 운영 제품의 PO/PM

여러 사용자군·권한·정책·데이터 정합성·운영 프로세스가 얽힌 제품. 본사·대리점·기사·CS 또는 설계사·운영·회계·보안처럼 같은 시스템을 다르게 쓰는 환경.

②

플랫폼·시스템 TPM/PM

ERP·CRM·CMS·SAP·모바일 앱처럼 여러 시스템이 연결되는 환경. 개발 책임자가 아니라 기술 제약을 제품 판단·일정·리스크 언어로 번역.

③

사업 모델 + 운영 시스템 + 도메인 모델링을 함께 보는 PO/PM

집닥 unit economics → 세인 현장 운영 → 토스 보험 플랫폼 + 수수료 환수 도메인 모델링. 기능 출시만 보는 역할보다 사업 구조와 운영 구조 + 도메인 모델을 같이 바꿔야 하는 역할.

④

외주·레거시 내재화 PM

ASP/SI 종료·이관, 단계적 내재화 로드맵, 협상 카드 누적, 운영 공백 방지. 도메인 모델 미스매치 같은 구조적 원인을 정량화해 내재화 결정으로 연결하는 패턴.

⑤

데이터 기반 의사결정 환경을 직접 여는 PM

분석 요청을 받고 쿼리 결과만 기다리는 게 아니라, 분석 가능해지는 전제(권한·계정·DB 접근·매칭 기준)를 PM이 직접 설계해야 하는 환경. 30일 골든타임 70%는 그 환경에서 나옴.

⑥

AI를 업무 시스템으로 넣는 PM

AI 기능이 아니라 벤더 협상·데이터 구조·보안 검수·비용 구조·운영 위생까지 PM이 직접 챙기는 영역. C06이 그 모델 (인켑 + AI UW 설계사 센터).

§ 06한 줄 요약 One-liner

"무엇을 만들었는가"보다 "어떤 패턴으로 일했는가"가 더 정확하게 저를 설명합니다.

도메인은 회계 → 인테리어 → 제조 → 보험으로 옮겼지만, 비용 대비 가치부터 보는 습관, 개발 안 하고 푸는 우선순위, 계약서·증적 기반 외주 협상, 결정 전 옵션 펼치기, 영역 경계 넘기 — 이 5개 패턴은 여러 도메인에서 일관되게 작동했습니다. 태니지먼트 강점 진단(평가+추진 강점 / 단순화·논리·효율·문제발견·달성·주도 6가지 재능)도 같은 패턴을 확인합니다. 그래서 다음 도메인에서도 같은 방식으로 작동합니다.
